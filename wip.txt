Про сертификаты.
Конфиг ссылается на /etc/nginx/certs/nginx.crt и nginx.key, но самих файлов пока нет.
Для локальной разработки нужен self-signed сертификат — по аналогии с тем, что ты уже делал в inception для WordPress.
Команда: openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
	-keyout nginx/certs/nginx.key \
	-out nginx/certs/nginx.crt \
	-subj "/C=ES/ST=Catalonia/L=Barcelona/O=42/CN=localhost"

* req -x509 — Говорит утилите, что мы хотим создать именно самоподписанный сертификат структуры X.509 (стандарт для веб-безопасности), а не просто запрос на подпись (CSR) для отправки в сторонний удостоверяющий центр.

* -nodes (читается как no-DES) — Этот флаг отключает шифрование закрытого ключа паролем. Если его не поставить, то каждый раз при перезапуске контейнера Docker, Nginx будет намертво зависать и требовать ввести пароль в консоли. Для автоматизации докера -nodes обязателен.

* -days 365 — Срок действия цифрового паспорта твоего сайта. Ровно через год он превратится в тыкву, но для защиты проекта в кампусе этого хватит с головой.

* -newkey rsa:2048 — Одновременно с сертификатом сгенерировать новый приватный ключ по алгоритму RSA длиной 2048 бит (золотой стандарт криптографии).

* -keyout и -out — Пути, куда физически сохранить приватный ключ (.key) и сам публичный сертификат (.crt).

* -subj — Поля субъекта (информация о владельце). Вот тут твоя напарница очень грамотно зашила локацию Барселоны и кампуса:

C=ES (Country — Испания)

ST=Catalonia (State/Region — Каталония)

L=Barcelona (Locality — Барселона)

O=42 (Organization — Школа 42)

CN=localhost (Common Name — доменное имя, для которого выпускается сертификат).

Браузер будет ругаться "небезопасное соединение" (потому что сертификат самоподписанный, а не от доверенного CA) — это нормально для dev-окружения, ты просто жмёшь "продолжить" один раз в браузере.

Чтобы не пришлось каждый раз на новом компе вручную запускать эту команду openssl, можно заставить Docker делать это автоматически при сборке контейнера Nginx, генерировать сертификат прямо на этапе сборки образа (RUN openssl... в Dockerfile).
Для этого мы завернем создание сертификатов прямо в Dockerfile для Nginx.
Но чем косяк такого подхода?
Образ собирается один раз. Это значит, что дата выпуска сертификата намертво «запекается» в этот неизменяемый образ. Если кто-то скачает собранный образ через полгода, у его сертификата останется всего половина срока жизни. А если вы развернете проект на продакшн-сервере, у вас у всех будет абсолютно один и тот же приватный ключ, что с точки зрения безопасности — полная лажа.

сделаем следующим образом (это Production-Ready решение уровня крепкого Senior DevOps инженера):
перенести генерацию на этап старта контейнера (Runtime) - entrypoint-скрипт, генерирующий сертификат при старте контейнера:
1) При первом make скрипт видит, что папка пустая, и генерирует свежий, уникальный для конкретной машины ключ.
2) При всех последующих рестартах (или если контейнер упал и поднялся), условие if [ ! -f ... ] проверяет наличие файлов, видит их и не тратит время на перегенерацию. Настоящее бережное отношение к ресурсам системы.

Это скрипт, который выполняется при каждом запуске контейнера (не при сборке образа), проверяет "а сертификат уже есть?" — если нет, генерирует, если есть — просто стартует nginx. Так сертификат живёт в volume на хосте (мы уже монтируем ./nginx/certs), генерируется один раз на конкретной машине разработчика и переиспользуется между перезапусками контейнера, но при этом процесс полностью автоматический для нового человека в команде — просто docker compose up, и всё само создастся.
Это тот же паттерн, что мы обсуждали для Prisma-миграций (пункт 3 от "подруги") — entrypoint-скрипт, который что-то готовит перед стартом основного процесса. Логично сделать оба таким образом, раз уж мы к этому пришли.

Разбор нескольких моментов в коде(речь о Dockerfile для nginx и скриптик entrypoint.sh)
1) exec nginx -g "daemon off;" — важная деталь, без которой у тебя будут проблемы с graceful shutdown.
Если просто вызвать nginx -g "daemon off;" без exec, эта команда запустится как дочерний процесс твоего shell-скрипта, и PID 1 в контейнере останется у самого shell'а.
Docker посылает SIGTERM именно PID 1 при остановке контейнера — если это shell, а не nginx, сигнал может не долететь до nginx корректно, и Docker будет ждать таймаут, а потом жёстко убьёт контейнер (SIGKILL). exec заменяет процесс shell'а процессом nginx в том же PID, так что сигналы приходят туда, куда надо.
Ещё раз:
- Когда Docker запускает контейнер, процесс, указанный в ENTRYPOINT, получает PID = 1 (главный процесс системы, аналог init в Linux).
Если написать просто nginx: Твой скрипт entrypoint.sh останется работать как PID 1. Когда ты пишешь docker-compose down (или make down), Docker посылает сигнал SIGTERM процессу PID 1. Но обычный bash/sh скрипт не умеет пересылать сигналы своим дочерним процессам! В итоге Nginx даже не узнает, что его хотят закрыть. Он будет работать, пока у Докера не кончится терпение (обычно 10 секунд), после чего Докер жестко пристрелит контейнер через SIGKILL. Для базы данных это означало бы потерю данных, для Nginx — битые логи.
- Что делает exec: В Си аналогом является системный вызов execve(). Он не создает новый процесс. Он берет текущий процесс PID 1 (наш скрипт) и полностью заменяет его код в памяти на код Nginx. Скрипт умирает, а Nginx становится владельцем PID 1. Теперь все сигналы SIGTERM прилетают прямо в Nginx, он успевает закрыть все сетевые соединения, дописать логи и корректно завершить работу (graceful shutdown) за доли секунды.

2) RUN apk add --no-cache openssl — образ nginx:alpine минималистичный, openssl там по умолчанию не установлен (хотя многие ожидают что он есть "как в обычном линуксе"). --no-cache — чтобы alpine не сохранял кэш пакетного менеджера внутри слоя образа, экономит место.

3) Права на entrypoint.sh через RUN chmod +x — та же логика, что и с UID-владением файлов, о которой мы говорили раньше: скрипт должен быть исполняемым внутри образа, это отдельное действие от простого копирования файла.
